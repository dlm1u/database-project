# Автоматизированная система логистики склада

---

##  О проекте
 
Данная база данных предназначена для автоматизации учета товаров на складе, управления заказами и контроля взаимодействия с клиентами, поставщиками и сотрудниками.

Система позволяет:
- контролировать остатки товаров  
- обрабатывать заказы  
- анализировать загрузку сотрудников  
- обеспечивать целостность данных на уровне БД  

---

## Архитектура проекта

Проектирование выполнено в три этапа:

1. **Концептуальная модель** — определение основных сущностей и связей  
2. **Логическая модель** — нормализация данных, определение ключей  
3. **Физическая модель** — выбор типов данных и ограничений  

> Все схемы доступны в директории `./schemes`

---

## Технологический стек

- **СУБД:** PostgreSQL  
- **Проектирование:** draw.io  
- **Тестирование:** pytest + psycopg2  

---

##  Структура базы данных

В базе реализовано 6 таблиц:

- `CLIENT` — информация о клиентах  
- `PRODUCT` — товары  
- `WAREHOUSE` — склады  
- `SUPPLIER` — поставщики  
- `EMPLOYEE` — сотрудники  
- `ORDER` — заказы  

Таблица `ORDER` является центральной и связывает все остальные сущности.

##  Дополнительный функционал

###  Представления

####  `order_full_info`
Представление, объединяющее данные из нескольких таблиц:
- клиент  
- товар  
- склад  
- сотрудник  

Позволяет:
- упростить сложные JOIN-запросы  
- получать полную информацию о заказе одним запросом  

Тип: **обычное представление**  
Причина: требуется актуальность данных в реальном времени  

---

####  `employee_workload_mv`
Материализованное представление, отображающее:
- общее количество заказов на сотрудника  
- количество заказов в обработке  

Тип: **materialized view**  
Причина: содержит агрегаты (COUNT), вычисление которых ресурсоёмко  

Обновление:
```sql
REFRESH MATERIALIZED VIEW employee_workload_mv;
```

##  Триггеры

### Резервирование товара
*   **Тип:** `BEFORE INSERT` (при создании заказа)
*   **Действие:** Проверяет наличие товара и уменьшает его количество на складе.
*   **Назначение:** Предотвращает создание заказов при недостаточном остатке.

### Возврат товара при отмене заказа
*   **Тип:** `AFTER UPDATE`
*   **Действие:** Возвращает товар на склад при смене статуса заказа на `cancelled`.
*   **Особенность:** Исключает повторное начисление товара за счет строгой проверки предыдущего статуса.

##  Процедуры и функции


* **get_order_total(order_id)** Возвращает общую стоимость заказа
* **get_employee_workload(emp_id)**  Возвращает количество активных заказов
* **update_order_status(id, status)**  Процедура смены статуса заказа
